Architecture Map

一句话架构：ECS 管状态，Feature 管贡献，Projection 管同步

框架层横向稳定，业务层纵向切片。业务不要碰 dirty、snapshot、registry key、full sync 这类机械细节。

框架负责机械工作

创建、汇总、校验、挂载、dirty mark、full sync、snapshot、flush、清理。

业务保留真实规则

Component 字段、Entity 初始化、Feature 拓扑、System 规则、Projection 映射、Node 表现。

边界越清楚，新增业务越少改框架。
Composition Root

GameScene 是组合根：创建 runtime，不写业务规则

GameScene.init() 把 world、EntityRuntime、ProjectionRuntime、EffectRuntime、ViewRegistry 和网络回调接起来。

init()
  -> createGameWorld()
  -> createEntityRuntime(world, registry.entityTypes())
  -> entityRuntime.bootstrapSingletons()
  -> createProjectionRuntime(registry.projections())
  -> createEffectRuntime()
  -> registry.setupAll(createFeatureRuntimeContext(...))
  -> create GameLoopPipeline(...)
  -> projectionRuntime.mark(world)
  -> projectionRuntime.sync(world)
组合根可以知道所有 runtime

这里是依赖汇合点，所以可以拿到 world、network、registry、projection、effect。

组合根不承载业务规则

命中、状态机、怪物生成、得分更新放在 session 或 feature system 中。

首次同步在 init 末尾

Feature setup 挂好 View 后，ProjectionRuntime 做一次初始 mark/sync。

文件：src/app/GameScene.ts

ECS Authority

EntityDefinition 描述创建契约，EntityRuntime 执行创建

Component 是权威状态。View node 只是表现，不保存规则事实。

defineEntity<MonsterEntityInput>({
  name: "monster",
  components: [MonsterComponent],
  cardinality: "many",
  initialize: (eid, input) => {
    MonsterComponent.monsterType[eid] = input.monsterType;
    MonsterComponent.visible[eid] = 0;
    MonsterComponent.spawnSeq[eid] = 0;
  },
});
cardinality: "one"

单例实体，启动时由 bootstrapSingletons() 创建。

cardinality: "many"

由 Feature setup 创建，常用于固定拓扑或对象池。

运行期策略

一般不 removeEntity，用状态字段复用槽位。

EntityRuntime 负责 addEntity、addComponent、初始化字段和单例缓存。业务不直接操作 bitecs 创建细节。

文件：framework/ecs/EntityDefinition.ts、EntityRuntime.ts

Projection

Projection 把持久可见状态映射到 View contract

System 只改 Component。Projection 负责把 Component 字段变化翻译成 View 方法调用。

const source = projectionSource("monster", MonsterComponent);

export const MonsterProjection = defineProjection<IMonsterNode>({
  name: "monster",
  components: [MonsterComponent],
  rows: [
    watch(source, ["monsterType", "spawnSeq"], "monster spawn", ({ eid, node }) => {
      node.spawn(MonsterComponent.monsterType[eid], MonsterComponent.spawnSeq[eid]);
    }),
    watch(source, ["posX", "posY"], "monster position", ({ eid, node }) => {
      node.setPosition(MonsterComponent.posX[eid], MonsterComponent.posY[eid]);
    }),
  ],
});
watch(...)

声明一组字段和一个 apply 函数。字段变化时才调用 apply。

View contract

Projection 调接口，Node 实现接口。规则层不 import Laya。

noProjection

规则字段可以被 watch 用于 dirty 分组，但不需要更新 View。

文件：framework/sync/ProjectionDefinition.ts、game/features/*/*Projection.ts
Dirty Mark

ProjectionRuntime 负责 snapshot、dirtyBits、fullSync 和 node 绑定

业务不维护脏标记。每帧统一比较 Component 字段，按变化 row 同步 View。

mount(projection, eid, node)
  -> nodes.set(eid, node)
  -> fullSync[eid] = 1

mark(world)
  -> query entities
  -> compare fields with snapshots
  -> dirtyBits[eid] = changed row bits

sync(world)
  -> find node by eid
  -> apply changed rows or full sync
  -> clear dirtyBits/fullSync
首次挂载为什么能显示

mount 会设置 fullSync[eid]，下一次 sync 会完整执行 rows。

为什么不会每帧全量刷 View

mark 只在字段值变化时设置 row bit。

为什么同一个 apply 不重复

Runtime 会跳过同一轮中已经执行过的相同 apply。

文件：framework/sync/ProjectionRuntime.ts
Effect

Effect 处理瞬时事实：reward、miss、一次性声音或动画

Projection 适合持久状态；Effect 适合“发生过一次”的事实。emit 只入队，帧末 flush。

const HitRewardEffect = defineEffect<HitReward>("hit.reward");

// system/session
ctx.effects.emit(HitRewardEffect, reward);

// feature setup
ctx.effects.on(HitRewardEffect, reward => {
  hudNode.showReward(reward);
});

// GameLoopPipeline.update
this._deps.effects?.flush();
typed definition

EffectDefinition 用对象身份匹配 handler，不靠全局字符串 EventBus。

pending queue

emit 写入队列，flush 统一分发。

clear on exit

world 退出时清空 handler 和 pending，避免旧场景收到回调。

文件：framework/sync/EffectDefinition.ts、EffectRuntime.ts
View Layer

View 层隔离 Laya：Node 负责表现，ViewRegistry 负责清理

Laya API 集中在具体 Node 和 framework/view 窄兼容层。规则、Entity、ProjectionDefinition 不依赖引擎对象。

文件职责
ViewRegistry.ts持有 destroyable 资源，场景退出时倒序销毁。
LayaRuntime.ts统一获取 window.Laya。
LayaLifecycle.ts统一 destroy、清 children、清 tween。
LayaLoader.ts资源加载和 Spine template Promise 缓存，失败时删除缓存以便重试。
LayaSpine.ts兼容不同 Spine 创建方式。

Frame Loop

每帧顺序决定边界：先规则，后表现

GameLoopPipeline 是运行期唯一主线。所有系统跑完后才统一做 dirty mark、view sync 和 effect flush。

state systems核心状态机、计时
network.update处理回包
feature systemsMonster/PerfHero 等功能系统
projection.mark比较 Component 快照
projection.sync调用 View contract
effects.flush派发瞬时事实
Laya render引擎表现
为什么 network 在中间

回包可以更新 ECS 状态或 emit Effect，但不直接操作 View。

为什么 sync 在最后

一帧内多个 system 可以合并成一次差量投影。

文件：src/app/GameLoopPipeline.ts
Lifecycle

一般只有一个 world：进场创建，退场整批销毁

ECS 的性能压力在运行期。初始化阶段可以用更清楚的封装；退出时清掉 runtime、snapshot、handler、View 和 world。

enter scene
  -> create world
  -> compile EntityRuntime / ProjectionRuntime / EffectRuntime
  -> bootstrap singleton entities
  -> Feature setup creates topology, pools, views
  -> initial projection sync

frame
  -> state systems -> network -> feature systems
  -> dirty mark -> view sync -> effect flush

exit scene
  -> clear network callbacks
  -> effectRuntime.clear()
  -> projectionRuntime.clear()
  -> viewRegistry.clear()
  -> entityRuntime.clear()
  -> drop world reference
运行期通常不 removeEntity

Shrew、Monster、PerfHero 用固定槽位或池，靠状态字段复用。

避免旧场景泄漏

异步资源、Effect handler、Projection node map、View destroy 都要在退出时清掉。

重新进入重新创建

新 world、新 runtime、新 snapshot，避免 eid 和旧状态交叉污染。

这也是当前测试重点：GameRuntimeLifecycle、GameScene lifecycle。
New Feature

新增 Feature 的标准路径：自己的目录内完成，只注册一行

框架层已经提供小型装配原语。业务切片保留真实规则，不继承 BaseFeature，不做自动扫描。

目录

src/game/features/foo/ 内放 Component、Entity、System、Projection、ViewContract、Node、Config。

Feature

用 defineFeature 声明 entities、projections、systems、setup。

注册

在 src/game/GameFeatures.ts 加一行显式 import 和数组项。

defineFeature({
  name: "foo",
  entities: [FooEntity],
  projections: [FooProjection],
  systems: [defineSystem("feature", "foo.state", fooSystem)],
  setup: ({ entities, mountPool }) => {
    const eids = createFooPool(entities, config);
    mountPool({ eids, projection: FooProjection, create: () => new FooNode() });
  },
});
新增业务主要复制 MonsterFeature 或 ShrewFeature 的形状。
Boundaries

边界清单：什么能做，什么不能越界

这些规则不是为了形式整洁，而是为了让 Feature 可复制、规则可测试、Laya 升级可控。

Component 是权威状态

View 只能展示，不保存游戏规则事实。

System 保持纯净

不 import Laya、Node、resource loader、registry。

Projection 走 contract

持久可见状态通过 *Projection.ts 同步。

Effect 传瞬时事实

reward、miss、声音、一次性动画不要塞进 Projection。

Feature 不导入别的内部文件

跨 Feature 走公开 index.ts 或 session。

业务不维护 dirty

dirtyBits、snapshot、fullSync、node map 都属于 framework/sync。

判断代码放哪：稳定机制放 framework；业务规则和拓扑放 game/features/<name>；跨 Feature 输入、网络回包编排放 game/session；Laya 应用壳放 app。

用边界降低长期改动成本。
Summary

最后记住这条链路：状态变了，框架把变化投到表现

新增业务时，不需要理解所有内部实现；先把自己的权威状态、规则、投影和装配声明清楚。

input / network callback
  -> game/session
  -> Feature system or domain helper
  -> Component
  -> ProjectionRuntime.mark(world)
  -> ProjectionRuntime.sync(world)
  -> View contract
  -> Laya Node

transient result
  -> EffectRuntime.emit(...)
  -> GameLoopPipeline.flush()
  -> view handler
