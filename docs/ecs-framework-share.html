ECS Basics

用 ShrewComponent 看懂 bitecs 的数据模型

bitecs 里没有“地鼠对象”。地鼠只是一个 eid，状态、血量、能不能点这些值，都放在 Component 的数组里。

Entity：shrewEid

创建一只地鼠时，`addEntity(world)` 返回一个数字。后面说“这只地鼠”，其实就是拿这个数字去查数据。

Component：一组字段数组

`ShrewComponent.actionState[shrewEid]` 是动作状态，`hp[shrewEid]` 是血量，`isClickable[shrewEid]` 是能不能点。

System：批量处理地鼠

`defineQuery([ShrewComponent, AnimationComponent])` 找到所有地鼠，再逐个 eid 推进 Wait、Up、Stand、Down、Dizzy。

对应代码：src/ecs/components/index.ts, src/ecs/world.ts, src/ecs/systems/ShrewStateSystem.ts
bitecs vs Unity DOTS

可以用 DOTS 的心智模型理解 bitecs，但实现重量不同

bitecs 更像轻量 SoA ECS：组件是 typed array，查询是 defineQuery，调度顺序由项目代码显式安排。

概念 bitecs 项目写法 Unity DOTS 心智对应 差异重点
Entity number eid Entity bitecs 实体就是数字，生命周期由 world 和 add/remove component 管。
Component defineComponent({ field: Types.f32 }) IComponentData bitecs 使用 SoA typed array，字段通过 Component.field[eid] 读写。
Query defineQuery([A, B, C]) EntityQuery / SystemAPI.Query 只表达“拥有哪些组件”，不内置复杂 Job 调度。
System 普通 TS 函数 ISystem / SystemBase 项目用 GameLoopPipeline 固定顺序，比引擎调度更直观。
Presentation binding + ViewNode Hybrid Renderer / Authoring Laya 节点不是权威状态，只是 dirty 后的投影对象。
项目里的 bitecs 四步 
const shrew = addEntity(world);
addComponent(world, ShrewComponent, shrew);
addComponent(world, AnimationComponent, shrew);

ShrewComponent.actionState[shrew] = ShrewAction.Wait;

const shrewQuery = defineQuery([ShrewComponent, AnimationComponent]);
const entities = shrewQuery(world);
和传统对象写法的区别

不是 `shrew.actionState = Wait`，而是 `ShrewComponent.actionState[shrewEid] = Wait`。System 拿到的是一批 eid，再用 eid 去读写各个 Component 字段数组。

DOTS 更完整，包含 Job、Burst、Baking 等生产体系；本项目使用 bitecs 是为了在 Web/Laya 客户端里获得清晰边界、可测试系统和低成本数据遍历。

项目关键词：defineComponent, defineQuery, GameLoopPipeline, SyncView
Architecture Boundary

分层边界：规则不能依赖 Laya，回包不能直接改 View

这句话说白了就是：外部事件先进适配器，适配器改 ECS 数据；画面只从 dirty binding 读数据，不反过来改规则。

谁能改规则？

ECS system 或 lifecycle helper。比如点击命中后，`hitDetectionSystem` 改 `ShrewComponent.hp/actionState/isClickable`。

View 做什么？

View 只接收 binding 投影过来的结果，比如 `ShrewNode.setAnimation(...)`。它不知道 protobuf，也不决定能不能命中。

为什么这么麻烦？

因为这样才好测试。规则不用启动 Laya，协议不用碰节点，画面问题也能沿着 dirty 链路查。

对应文档：docs/architecture.md, docs/laya-rules.md
Runtime Loop

每帧系统顺序决定了“先规则，后表现”

GameLoopPipeline.update(deltaSec) 显式安排系统顺序，避免隐式事件链让调试变困难。

animationTimerSystem动画进度
shrewStateSystem地鼠 FSM
sceneCycleSystem地图轮换
hammerSystem锤子冷却
network.update超时检查
dirtyMarkSystem字段 diff
syncView.sync投影到 Laya
deltaSec 是事实时间

状态机和动画计时都使用 GameScene.update(deltaSec) 传入的真实帧间隔，不把规则绑死在固定 60fps 上。

dirty 在系统之后统一计算

系统只改 component。DirtyMarkSystem 比较本帧和上一帧快照，再让 SyncView 决定哪些节点需要更新。

GameScene.update(delta)
  -> GameLoopPipeline.update(delta)
  -> animationTimerSystem
  -> shrewStateSystem
  -> sceneCycleSystem
  -> hammerSystem
  -> network.update
  -> perfHeroSystem
  -> dirtyMarkSystem
  -> syncView.sync
对应代码：src/view/GameLoopPipeline.ts
Protocol Boundary

protobuf 网络边界：二进制收发，不让协议细节扩散到 View

客户端只认业务类型和 Uint8Array。真正的 proto 文件和协议号来自兄弟 Go 服务，View 层不需要知道 wire 格式。

请求怎么出去

`KickInputAdapter` 只组业务请求；`KickSocket.sendKick()` 加 seqId，`KickProtoCodec` 包成 protobuf 二进制。

真实服务器在哪里

真实模式走 `WebSocketTransport` 到 `../GoServerActorFsm`。本地 mock 只是替身，也必须走 protobuf 编解码。

回包怎么回来

回包按 seqId 匹配 pending，再由 `NetworkAdapter.onResponse` 交给 `hitResponseSystem`，不是直接改 View。

对应代码：src/network/KickSocket.ts, src/network/KickProtoCodec.ts, docs/protocol.md
State Machine

状态机：少状态、明语义、统一重置入口

地鼠状态已经收敛到 Wait / Up / Stand / Down / Dizzy。自然离场和命中离场语义分开，避免 Down 期间再次被击中的歧义。

自然循环

Wait 随机等待，Up 播出洞动画，Stand 开启可点击，Down 播入洞动画，最后重置回 Wait。

命中循环

HitDetectionSystem 判断命中后调用 startShrewDizzyHold，Dizzy 短暂停留结束后直接重置到 Wait。

测试重点

每条状态边都能用单测重放，尤其是 deltaSec 计时、蓝鼠双击、Dizzy 重置等边界。

对应代码：src/ecs/systems/ShrewStateSystem.ts, src/ecs/ShrewLifecycle.ts
Dirty Mark / Binding

dirty mark 的本质：把字段变化翻译成 view 更新 bit

System 只改 Component。DirtyMarkSystem 用上一帧快照做 diff，把变化写进 DirtyComponent；SyncView 再按 bit 调用对应 binding。

第一次同步为什么全量？

`DirtySchemaRunner` 第一次看到 eid 时没有 snapshot，会填充快照并写 `channel.allBits`，让 view 拿到初始状态。

之后为什么差量？

后续每帧只比较 DirtyAspect 声明的字段；字段值不同才 OR 对应 bit，并更新 snapshot，SyncView 同步后清空 bit。

对应代码：src/ecs/dirty/DirtySchemaRunner.ts, src/ecs/dirty/aspects/SceneDirtyAspect.ts, src/binding/SyncView.ts
Scene Dirty Example

例子：SceneComponent 场景从 Meadow 切到 Ship

这页用一个场景单例讲透 dirty 链路：谁写数据、谁发现变化、dirty bit 写到哪里、最后哪个 Laya 方法被调用。

1. System 写权威状态 
SceneComponent.currentMap[sceneEid] = nextMap;
SceneComponent.sceneTimer[sceneEid] = 0;
SceneComponent.transitioning[sceneEid] = 1;
DirtyComponent.forceFullSync[sceneEid] = 1;
2. DirtyAspect 声明字段到 bit 
mark(BIT_SCENE_MAP, "当前地图", [
  field("SceneComponent.currentMap", SceneComponent.currentMap),
], "SceneLayer.setMap")
3. Binding 消费 bit 
if (forceFull || (dirtyBits & BIT_SCENE_MAP)) {
  node.switchScene(SceneComponent.currentMap[eid]);
}

Shrew Dirty Example

例子：地鼠从 Wait 变成 Up，画面为什么会动

状态机只改 ECS 数据。dirty 系统发现 `ShrewComponent` 和 `AnimationComponent` 变了，binding 才去叫 `ShrewNode` 播出洞表现。

1. 状态机写数据 
ShrewComponent.actionState[eid] = ShrewAction.Up;
AnimationComponent.animType[eid] = AnimType.Up;
AnimationComponent.progress[eid] = 0;
AnimationComponent.duration[eid] = timing.upDurationSec;
2. DirtyAspect 找到变化 
BIT_SHREW_ACTION -> ShrewComponent.actionState
BIT_ANIM_TYPE    -> AnimationComponent.animType
BIT_ANIM_PROGRESS -> AnimationComponent.progress
3. Binding 更新画面 
node.setAnimation(
  ShrewComponent.actionState[eid],
  AnimationComponent.animType[eid],
  AnimationComponent.progress[eid],
);
对应代码：src/ecs/systems/ShrewStateSystem.ts, src/ecs/dirty/aspects/ShrewDirtyAspect.ts, src/binding/ShrewViewBinding.ts
TDD Workflow

TDD：先证明问题，再改代码，最后整理和提交

这里的 TDD 不只是写测试。它还包括改前确认、改后验证、Refactor 检查，以及用 git 把这次工作讲清楚。

Red：先让问题可见

先跑最小相关测试。如果没有测试，就补一个能复现行为的测试。比如状态机改动先看 `ShrewStateSystem.test.ts`，dirty 改动先看 `DirtyMarkSystem.test.ts`。

Green：只做够用的改动

改到测试刚好通过，不顺手重构一大片。比如新增字段，先让 Component、System、DirtyAspect、Binding 这条链路通。

Refactor：整理边界和命名

测试绿了以后再检查：规则有没有跑进 View？View 有没有改 ECS？dirty bit 名字和字段是否对齐？重复逻辑能不能收进 helper？这一步是让代码下次还能改。

一次完整 git 流程 
# 1. 改前看工作区，别碰用户无关改动
git status --short

# 2. Red / Green 验证
npm test -- --run src/tests/ecs/ShrewStateSystem.test.ts
npm test -- --run src/tests/ecs/DirtyMarkSystem.test.ts src/tests/binding/ShrewViewBinding.test.ts

# 3. 类型检查，运行时改动再跑 debug
npx tsc --noEmit
npm run debug:ready

# 4. 只暂存本次相关文件
git diff
git add src/ecs/... src/binding/... src/view/... src/tests/...

# 5. commit 写清楚三段
git commit -m "feat: add score milestone celebration" \
  -m "TDD Red: ..." \
  -m "TDD Green: ..." \
  -m "TDD Refactor: ..."
对应文档：docs/test-guide.md
UI Debug Tips

界面调试技巧：把视觉问题拆成坐标、层级、资源、生命周期

Laya 表现层允许使用 Laya.*，但它只应该表现 ECS 状态。界面 bug 要沿着 owner 和投影链路定位。

坐标

输入用设计分辨率比例，洞位配置用 posXRatio/posYRatio，ShrewNode 处理 Cocos Y-up 到 Laya Y-down。

层级

SceneLayer 管背景、cover 和过渡遮罩；洞位 zOrder 与地图行、cover 资源一起调。

资源

Atlas 逻辑名集中在 AtlasConfig，getFrameTexture 按 url 后缀找子帧，rotated=true 需要手动补偿。

生命周期

timer、tween、event、loader callback、registry 引用都要有 owner 和 teardown。

对应文档：docs/laya-rules.md
Performance Stress

压测调试：先分清 ECS 压力和 Laya 渲染压力

perf 英雄链路专门用来测 Spine/Skeleton、dirty binding、节点池和长时间运行压力。

压测入口

http://localhost:8080/debug-tsc.html?perf=1&heroes=200，可用 heroes/heroCount 控制数量，shrewFast=0 可关闭地鼠加速。

观察指标

FPS、FrameTime、DrawCall、Triangle、Sprite2DCount、JS Heap、GPUMemory、AllTexture、GPUBuffer 一起记录。

常见结论

200 个 Spine 的主要压力通常不在 bitecs，而在 Laya Skeleton 每帧骨骼、slot、mesh 更新和渲染。

对应文档：docs/performance-tuning.md
Implementation Checklist

落地清单：用“每 1000 分祝贺动画”举例

需求听起来是表现需求，但不要直接在 View 里判断分数。先定谁是权威状态，再让 dirty binding 把结果推到动画节点。

1. 规则层先定清楚

假设分数来自 `PlayerComponent.score`。新增 `lastCelebratedK`、`celebrationVisible`、`celebrationTimerSec`。当分数从 1900 到 2050，系统只触发 2000 这一档。

2. dirty 链路写完整

`PlayerComponent.celebrationVisible -> BIT_PLAYER_CELEBRATION -> DirtyComponent.playerDirty -> playerViewBinding -> PlayerHUD.showCelebration()`。

3. 表现层只播动画

`PlayerHUD` 或新的 `CelebrationNode` 放在屏幕正下方，显示 3 秒。动画怎么 tween 是 View 的事，什么时候出现由 ECS 字段决定。

4. 测试先覆盖边界

测 999 不弹、1000 弹、1999 不重复弹、2000 再弹、3 秒后隐藏。binding 测 visible 变化会调用节点方法。

对应清单：Component 字段 -> System 规则 -> DirtyFlags/DirtyAspect -> Binding -> ViewNode -> 测试 -> debug:ready。
