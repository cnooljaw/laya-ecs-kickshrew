Architecture Map

一句话架构：ECS 管状态，Feature 写业务模块，Projection 管同步

框架层负责固定流程，业务层按功能切目录。业务不要碰 dirty、snapshot、registry key、full sync 这类机械细节。

框架负责固定流程

创建、整理、校验、挂载、dirty mark、full sync、snapshot、flush、清理。

业务写自己的模块

Component 字段、Entity 初始化、Feature 拓扑、System 规则、Projection 映射、Node 表现。

边界越清楚，新增业务越少改框架。
Composition First

先理解组合优于继承：游戏对象经常不是一棵类树

继承适合稳定的“是什么”层级；游戏运行期更常见的是类型、状态、Buff、动画、碰撞、网络和池化能力交叉组合。

继承写法很快膨胀 
class Shrew {}
class RedShrew extends Shrew {}
class BlueHatShrew extends RedShrew {}
class FastBlueHatShrew extends BlueHatShrew {}
class DizzyFastBlueHatShrew extends FastBlueHatShrew {}
组合写法把变化拆开 
eid = create shrew
ShrewComponent.shrewType[eid] = Blue
ShrewComponent.hasHat[eid] = 1
ShrewComponent.actionState[eid] = Stand
SpeedComponent.value[eid] = fast
设计判断：变化维度多且可叠加时，优先组合。
Why ECS

为什么游戏运行期常用 ECS，而不是只用 MVVM

MVVM 擅长表单、页面状态和低频 UI 绑定；游戏主循环要处理大量实体、计时、碰撞、池化、动画状态和网络回包合流。

关注点MVVM 更适合ECS 更适合
更新频率用户操作驱动、低频界面刷新每帧批量推进大量实体
状态来源ViewModel 贴近界面结构Component 保存游戏权威事实
规则组织围绕页面和控件组织围绕 System 批量处理规则
表现同步数据绑定直接驱动 UIProjection 差量同步到 View contract
本项目的选择

Laya Node 是投影目标，不是状态权威。金币、地鼠状态、怪物生命周期都先落到 Component。

不是否定 MVVM

HUD、设置面板、背包列表这类纯 UI 仍然可以借鉴 MVVM；但战斗/玩法运行期要有统一帧循环。

关键收益

规则可测，View 可替换，运行期合批同步，初始化阶段允许更厚的装配封装。

一句话：游戏运行期先要稳定的事实和帧顺序，再谈界面表达。
Startup Wiring

GameScene 是启动装配入口：创建 runtime，不写业务规则

GameScene.init() 把 world、EntityRuntime、ProjectionRuntime、EffectRuntime、ViewRegistry 和网络回调接起来。

init()
  -> createGameWorld()
  -> createEntityRuntime(world, registry.entityTypes())
  -> entityRuntime.bootstrapSingletons()
  -> createProjectionRuntime(registry.projections())
  -> createEffectRuntime()
  -> registry.setupAll(createFeatureRuntimeContext(...))
  -> create GameLoopPipeline(...)
  -> projectionRuntime.mark(world)
  -> projectionRuntime.sync(world)
这里负责接线

这里是依赖汇合点，所以可以拿到 world、network、registry、projection、effect。

这里不写玩法规则

命中、状态机、怪物生成、得分更新放在 session 或 feature system 中。

首次同步在 init 末尾

Feature setup 挂好 View 后，ProjectionRuntime 做一次初始 mark/sync。

Projection

Projection 把持久可见状态映射到 View contract

System 只改 Component。Projection 负责把 Component 字段变化翻译成 View 方法调用。

const source = projectionSource("monster", MonsterComponent);

export const MonsterProjection = defineProjection<IMonsterNode>({
  name: "monster",
  components: [MonsterComponent],
  rows: [
    watch(source, ["monsterType", "spawnSeq"], "monster spawn", ({ eid, node }) => {
      node.spawn(MonsterComponent.monsterType[eid], MonsterComponent.spawnSeq[eid]);
    }),
    watch(source, ["posX", "posY"], "monster position", ({ eid, node }) => {
      node.setPosition(MonsterComponent.posX[eid], MonsterComponent.posY[eid]);
    }),
  ],
});
watch(...)

声明一组字段和一个 apply 函数。字段变化时才调用 apply。

View contract

Projection 调接口，Node 实现接口。规则层不 import Laya。

noProjection

规则字段可以被 watch 用于 dirty 分组，但不需要更新 View。

文件：framework/sync/ProjectionDefinition.ts、game/features/*/*Projection.ts
PlayerHud Example

例子：金币变化从回包到 HUD 的完整生命周期

服务端回包写入 PlayerComponent.money 后，不直接改 HUD。下一帧由 mark 比较快照并标 dirty，再由 sync 调用 PlayerHUD.setMoney。

1. setup 挂载 HUD 
mountOne({
  eid: playerEid,
  projection: PlayerProjection,
  create: () => new PlayerHUD(),
});
// mount 后设置 fullSync，首次 sync 会完整刷新
2. 回包写权威状态 
applyPlayerKickResponse(playerEid, response)
const old = PlayerComponent.money[playerEid];
PlayerComponent.money[playerEid] += response.money;
3. mark 标记 dirty 
ProjectionRuntime.mark(world)
  -> snapshot.money: old -> new
  -> dirtyBits[playerEid] |= money row
4. sync 更新 HUD 并清理 
watch(source, ["money"], "player money", ({ eid, node }) => {
  node.setMoney(PlayerComponent.money[eid]);
});
// sync 后清 dirty/fullSync，snapshot 更新为新值

Effect

Effect 是发布-订阅：一次性事实发出去，订阅方自己处理

Projection 同步持久状态；Effect 发布“刚发生过”的事实。发布方不关心谁处理，订阅方不反查业务系统，帧末统一 flush。

const HitRewardEffect = defineEffect<HitReward>("hit.reward");

// 订阅：PlayerHudFeature 关心奖励飘字
ctx.effects.on(HitRewardEffect, reward => {
  hitEffectNode.showReward(reward);
});

// 发布：KickResponseFlow 只发布事实
effects.emit(HitRewardEffect, reward);

// 帧末：统一分发并清空 pending
const current = pending;
pending = [];
for (const effect of current) dispatch(effect);
定义主题

EffectDefinition 是订阅主题，用对象身份匹配，不靠全局字符串 EventBus。

订阅方

PlayerHudFeature 订阅 HitRewardEffect，只负责飘字、miss 提示、声音这类表现。

发布方

KickResponseFlow 发布奖励事实，不 import HUD 节点，也不关心有几个订阅方。

退出清理

world 退出时 clear() 清空 handler 和 pending，避免旧场景收到回调。

不是持久状态

奖励飘字、miss 提示、一次性声音用 Effect；金币数值仍然写 Component。

队列生命周期

emit 只 pending.push；flush 先 pending = []，再分发本轮事件。

文件：framework/sync/EffectDefinition.ts、EffectRuntime.ts
View Layer

View 层隔离 Laya：Node 负责表现，ViewRegistry 负责清理

Laya API 集中在具体 Node 和 framework/view 窄兼容层。规则、Entity、ProjectionDefinition 不依赖引擎对象。

Protocol Boundary

protobuf 网络边界：二进制收发，不让协议细节扩散到 View

当前代码里，View 不读 proto；网络回包进入 session，再更新 ECS 或 emit Effect。协议细节停在 network/session 边界内。

请求怎么出去

KickInputController 只组业务请求；KickSocket.sendKick() 加 seqId，KickProtoCodec 包成 protobuf 二进制。

回包怎么回来

回包按 seqId 匹配 pending，再由 NetworkAdapter.onResponse 交给 routeKickResponse。

边界是什么

网络层不直接改 View；View 不知道 wire 格式。

对应代码：src/network/KickSocket.ts、KickProtoCodec.ts、src/game/session/KickResponseFlow.ts
State Machine

状态机：少状态、明语义、统一重置入口

state systems 放在每帧前半段：规则先决定状态，表现后面由 Projection 统一同步。命中时还要区分本地即时反馈和服务端回包权威。

自然循环

Wait 随机等待，Up 播出洞，Stand 开启可点击，Down 回洞，最后重置回 Wait。

Dizzy 的权威动作

触摸命中后，hitDetectionSystem 立刻 hp-- 并调用 startShrewDizzyHold。这是本地即时表现的权威动作。

hitResponse 的职责

服务端回包进入 routeKickResponse，更新金币、怒气、锤子和雷电，emit 奖励 Effect；它不再改 Shrew 的 Dizzy 状态。

先后顺序

touch -> local hit -> Dizzy -> send request；response -> score/resource/effect。这样命中反馈不等待网络。

对应代码：src/game/features/shrew/ShrewStateSystem.ts、src/tests/ecs/ShrewStateSystem.test.ts
Lifecycle

一般只有一个 world：进场创建，退场整批销毁

ECS 的性能压力在运行期。初始化阶段可以用更清楚的封装；退出时清掉 runtime、snapshot、handler、View 和 world。

enter scene
  -> create world
  -> compile EntityRuntime / ProjectionRuntime / EffectRuntime
  -> bootstrap singleton entities
  -> Feature setup creates topology, pools, views
  -> initial projection sync

frame
  -> state systems -> network -> feature systems
  -> dirty mark -> view sync -> effect flush

exit scene
  -> clear network callbacks
  -> effectRuntime.clear()
  -> projectionRuntime.clear()
  -> viewRegistry.clear()
  -> entityRuntime.clear()
  -> drop world reference
运行期通常不 removeEntity

Shrew、Monster、PerfHero 用固定槽位或池，靠状态字段复用。

避免旧场景泄漏

异步资源、Effect handler、Projection node map、View destroy 都要在退出时清掉。

重新进入重新创建

新 world、新 runtime、新 snapshot，避免 eid 和旧状态交叉污染。

这也是当前测试重点：GameRuntimeLifecycle、GameScene lifecycle。

Boundaries

边界清单：什么能做，什么不能越界

这些规则不是为了形式整洁，而是为了让 Feature 可复制、规则可测试、Laya 升级可控。

Component 是权威状态

View 只能展示，不保存游戏规则事实。

System 保持纯净

不 import Laya、Node、resource loader、registry。

Projection 走 contract

持久可见状态通过 *Projection.ts 同步。

Effect 传瞬时事实

reward、miss、声音、一次性动画不要塞进 Projection。

Feature 不导入别的内部文件

跨 Feature 走公开 index.ts 或 session。

业务不维护 dirty

dirtyBits、snapshot、fullSync、node map 都属于 framework/sync。

判断代码放哪：稳定机制放 framework；业务规则和拓扑放 game/features/<name>；跨 Feature 输入、网络回包编排放 game/session；Laya 应用壳放 app。

用边界降低长期改动成本。
