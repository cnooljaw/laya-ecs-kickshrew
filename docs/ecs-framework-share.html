ECS Intro

ECS 是什么：把“对象”拆成数据和规则

ECS 不是为了把代码写复杂。它的目的很朴素：状态统一放在 Component，规则统一放在 System，画面只是状态的投影。

Entity 是编号

`eid` 只是一个数字。比如一只地鼠、一个洞位、玩家 HUD、场景单例，都是 world 里的 entity。

Component 是数据表

`ShrewComponent.hp[eid]`、`SceneComponent.currentMap[eid]` 这种写法，本质就是用 eid 去数组里取字段。

System 是规则

状态机、命中、场景轮换、锤子冷却都在 system 里跑。System 改数据，不直接改 Laya 节点。

好处是边界清楚

规则可以单测，View 不决定玩法，网络回包不直接碰画面，dirty binding 只同步发生变化的字段。

一句话：Entity 定位一份数据，Component 保存字段，System 按规则改字段。
world.ts

用 world.ts 看我们怎么搭出地鼠 world

`createGameWorld()` 只创建空世界。真正的游戏状态，是后面把单例、洞位、地鼠这些 entity 一个个加进去。

1. 单例 entity

`createSingletonEntities(world)` 创建 hammer、combo、scene、player、network。它们不是全局变量，而是 world 里的固定 entity。

2. 洞位 entity

`createHoleEntities(world, MapType.Meadow)` 创建 9 个洞，记录行列、比例坐标、zIndex，以及 `shrewEid` 关联。

3. 地鼠 entity

`createShrewEntity(world, shrewType, mapType)` 给地鼠挂 `ShrewComponent`、`AnimationComponent`、`DirtyComponent`，然后用 `resetShrewForNextCycle` 初始化下一轮等待。

对应代码：src/ecs/world.ts, src/view/GameScene.ts
World Lifecycle

world 的生命周期：一局游戏一个 world，不让 eid 一直涨

这套原型不是每次地鼠出洞都创建新 entity。9 个洞和 9 只地鼠会被反复复用，只重置 Component 字段。

地鼠为什么不一直 addEntity

出洞、回洞、眩晕、下一轮，都只是改 `ShrewComponent` 和 `AnimationComponent` 字段。这样 eid 数量稳定，dirty snapshot 也稳定。

一局结束怎么处理

`GameScene.destroy()` 停止循环，销毁网络，`ViewRegistry.clear()` 解绑并销毁节点，最后让 scene/world 引用随对象释放。

如果真的要动态删除

先 unregister binding 和销毁 ViewNode，再 `removeEntity(world, eid)`。同时清掉这类 entity 的 dirty snapshot，避免以后复用 eid 时读到旧快照。

对应代码：src/view/GameScene.ts, src/view/ViewRegistry.ts, src/ecs/world.ts
ECS Basics

用 ShrewComponent 看懂 bitecs 的数据模型

bitecs 里没有“地鼠对象”。地鼠只是一个 eid，状态、血量、能不能点这些值，都放在 Component 的数组里。

Entity：shrewEid

创建一只地鼠时，`addEntity(world)` 返回一个数字。后面说“这只地鼠”，其实就是拿这个数字去查数据。

Component：一组字段数组

`ShrewComponent.actionState[shrewEid]` 是动作状态，`hp[shrewEid]` 是血量，`isClickable[shrewEid]` 是能不能点。

System：批量处理地鼠

`defineQuery([ShrewComponent, AnimationComponent])` 找到所有地鼠，再逐个 eid 推进 Wait、Up、Stand、Down、Dizzy。

对应代码：src/ecs/components/index.ts, src/ecs/world.ts, src/ecs/systems/ShrewStateSystem.ts
Architecture Boundary

分层边界：规则不能依赖 Laya，回包不能直接改 View

这句话说白了就是：外部事件先进适配器，适配器改 ECS 数据；画面只从 dirty binding 读数据，不反过来改规则。

Adapter 是翻译员

比如 `KickInputAdapter` 把 Laya 坐标转成比例坐标，调用 `hitDetectionSystem`，再把命中结果组装成 `KickRequest`。它不负责动画细节。

Entity 删除怎么做

先通过 `ViewRegistry`/binding unregister 掉节点引用并销毁节点，再 `removeEntity(world, eid)`。如果以后有动态删除，还要清理 dirty snapshot，避免旧 eid 快照干扰。

为什么这么麻烦？

因为这样才好测试。规则不用启动 Laya，协议不用碰节点，画面问题也能沿着 dirty 链路查。

对应文档：docs/architecture.md, docs/laya-rules.md
Runtime Loop

每帧系统顺序决定了“先规则，后表现”

GameLoopPipeline.update(deltaSec) 显式安排系统顺序，避免隐式事件链让调试变困难。

animationTimerSystem动画进度
shrewStateSystem地鼠 FSM
sceneCycleSystem地图轮换
hammerSystem锤子冷却
network.update超时检查
dirtyMarkSystem字段 diff
syncView.sync投影到 Laya
deltaSec 是事实时间

状态机和动画计时都使用 GameScene.update(deltaSec) 传入的真实帧间隔，不把规则绑死在固定 60fps 上。

dirty 在系统之后统一计算

系统只改 component。DirtyMarkSystem 比较本帧和上一帧快照，再让 SyncView 决定哪些节点需要更新。

GameScene.update(delta)
  -> GameLoopPipeline.update(delta)
  -> animationTimerSystem
  -> shrewStateSystem
  -> sceneCycleSystem
  -> hammerSystem
  -> network.update
  -> perfHeroSystem
  -> dirtyMarkSystem
  -> syncView.sync
对应代码：src/view/GameLoopPipeline.ts
Protocol Boundary

protobuf 网络边界：二进制收发，不让协议细节扩散到 View

客户端只认业务类型和 Uint8Array。真正的 proto 文件和协议号来自兄弟 Go 服务，View 层不需要知道 wire 格式。

请求怎么出去

`KickInputAdapter` 只组业务请求；`KickSocket.sendKick()` 加 seqId，`KickProtoCodec` 包成 protobuf 二进制。

真实服务器在哪里

真实模式走 `WebSocketTransport` 到 `../GoServerActorFsm`。本地 mock 只是替身，也必须走 protobuf 编解码。

回包怎么回来

回包按 seqId 匹配 pending，再由 `NetworkAdapter.onResponse` 交给 `hitResponseSystem`，不是直接改 View。

对应代码：src/network/KickSocket.ts, src/network/KickProtoCodec.ts, docs/protocol.md
State Machine

状态机：少状态、明语义、统一重置入口

地鼠状态已经收敛到 Wait / Up / Stand / Down / Dizzy。自然离场和命中离场语义分开，避免 Down 期间再次被击中的歧义。

自然循环

Wait 随机等待，Up 播出洞动画，Stand 开启可点击，Down 播入洞动画，最后重置回 Wait。

命中循环

HitDetectionSystem 判断命中后调用 startShrewDizzyHold，Dizzy 短暂停留结束后直接重置到 Wait。

测试重点

每条状态边都能用单测重放，尤其是 deltaSec 计时、蓝鼠双击、Dizzy 重置等边界。

对应代码：src/ecs/systems/ShrewStateSystem.ts, src/ecs/ShrewLifecycle.ts
Dirty Mark / Binding

dirty mark 的本质：把字段变化翻译成 view 更新 bit

System 只改 Component。DirtyMarkSystem 用上一帧快照做 diff，把变化写进 DirtyComponent；binding 再用同一张 ViewRules 表执行 view apply。

第一次同步为什么全量？

`DirtySchemaRunner` 第一次看到 eid 时没有 snapshot，会填充快照并写 `channel.allBits`，让 view 拿到初始状态。

之后为什么差量？

后续每帧只比较 DirtyAspect 或表格式 rules 声明的字段；字段值不同才 OR 对应 bit，并更新 snapshot，SyncView 同步后清空 bit。

对应代码：src/ecs/dirty/DirtySchemaRunner.ts, src/sync/rules/*ViewRules.ts, src/binding/SyncView.ts
Snapshot Diff

Snapshot 前后对比：dirty 是怎么被算出来的

DirtyMarkSystem 不是凭感觉标脏。它给每个 eid 留一份上一帧快照，这一帧再读一次 Component 字段，哪个字段不同就打开哪个 bit。

上一帧 snapshot 
eid = 17
{
  "ShrewComponent.actionState": Wait,
  "AnimationComponent.animType": Idle,
  "AnimationComponent.progress": 0,
  "AnimationComponent.duration": 0
}
这一帧 Component 数组 
ShrewComponent.actionState[17] = Up
AnimationComponent.animType[17] = Up
AnimationComponent.progress[17] = 0
AnimationComponent.duration[17] = 0.31
比较结果 
actionState: Wait -> Up       => BIT_SHREW_ACTION
animType:    Idle -> Up       => BIT_ANIM_TYPE
duration:    0 -> 0.31        => BIT_ANIM_DURATION
progress:    0 -> 0           => 不标
对应代码：src/ecs/dirty/DirtySchemaRunner.ts, src/ecs/dirty/aspects/ShrewDirtyAspect.ts
Scene Dirty Example

例子：SceneComponent 场景从 Meadow 切到 Ship

这页用一个场景单例讲透传统 dirty 链路：谁写数据、谁发现变化、dirty bit 写到哪里、binding 如何消费 bit。

1. System 写权威状态 
SceneComponent.currentMap[sceneEid] = nextMap;
SceneComponent.sceneTimer[sceneEid] = 0;
SceneComponent.transitioning[sceneEid] = 1;
DirtyComponent.forceFullSync[sceneEid] = 1;
2. ViewRules 声明字段到 bit 
rule(BIT_SCENE_MAP, "当前地图",
    ["currentMap"], applySwitchScene)
rule(BIT_SCENE_TIMER, "场景计时器",
    ["sceneTimer"], noView)
3. apply 函数消费 bit 
function applySwitchScene({ eid, node }) {
  node.switchScene(SceneComponent.currentMap[eid]);
}

Shrew Dirty Example

例子：地鼠从 Wait 变成 Up，画面为什么会动

状态机只改 ECS 数据。dirty 和 binding 共用 `*ViewRules` 表，字段变化负责开 bit，apply 函数负责更新节点。

1. 状态机写数据 
ShrewComponent.actionState[eid] = ShrewAction.Up;
AnimationComponent.animType[eid] = AnimType.Up;
AnimationComponent.progress[eid] = 0;
AnimationComponent.duration[eid] = timing.upDurationSec;
2. 表格式 rules 声明变化 
rule(BIT_SHREW_ACTION, "动作状态",
    ["actionState"], applyAnimation)
rule(BIT_ANIM_PROGRESS, "动画进度",
    ["progress"], applyAnimation)
3. apply 函数更新画面 
function applyAnimation({ eid, node }) {
  node.setAnimation(
  ShrewComponent.actionState[eid],
  AnimationComponent.animType[eid],
  AnimationComponent.progress[eid],
  );
}
对应代码：src/ecs/systems/ShrewStateSystem.ts, src/sync/rules/ShrewViewRules.ts, src/sync/rules/ViewBindingRule.ts
TDD Workflow

TDD：先证明问题，再改代码，最后整理和提交

这里的 TDD 不只是写测试。它还包括改前确认、改后验证、Refactor 检查，以及用 git 把这次工作讲清楚。

Red：先让问题可见

先跑最小相关测试。如果没有测试，就补一个能复现行为的测试。比如状态机改动先看 `ShrewStateSystem.test.ts`，dirty 改动先看 `DirtyMarkSystem.test.ts`。

Green：只做够用的改动

改到测试刚好通过，不顺手重构一大片。比如新增字段，先让 Component、System、DirtyAspect、Binding 这条链路通。

Refactor：整理边界和命名

测试绿了以后再检查：规则有没有跑进 View？View 有没有改 ECS？dirty bit 名字和字段是否对齐？重复逻辑能不能收进 helper？这一步是让代码下次还能改。

一次完整 git 流程 
# 1. 改前看工作区，别碰用户无关改动
git status --short

# 2. Red / Green 验证
npm test -- --run src/tests/ecs/ShrewStateSystem.test.ts
npm test -- --run src/tests/ecs/DirtyMarkSystem.test.ts src/tests/binding/ShrewViewBinding.test.ts

# 3. 类型检查，运行时改动再跑 debug
npx tsc --noEmit
npm run debug:ready

# 4. 只暂存本次相关文件
git diff
git add src/ecs/... src/binding/... src/view/... src/tests/...

# 5. commit 写清楚三段
git commit -m "feat: add score milestone celebration" \
  -m "TDD Red: ..." \
  -m "TDD Green: ..." \
  -m "TDD Refactor: ..."
对应文档：docs/test-guide.md
Performance Stress

压测调试：先分清 ECS 压力和 Laya 渲染压力

perf 英雄链路专门用来测 Spine/Skeleton、dirty binding、节点池和长时间运行压力。

压测入口

http://localhost:8080/debug-tsc.html?perf=1&heroes=200，可用 heroes/heroCount 控制数量，shrewFast=0 可关闭地鼠加速。

观察指标

FPS、FrameTime、DrawCall、Triangle、Sprite2DCount、JS Heap、GPUMemory、AllTexture、GPUBuffer 一起记录。

常见结论

200 个 Spine 的主要压力通常不在 bitecs，而在 Laya Skeleton 每帧骨骼、slot、mesh 更新和渲染。

对应文档：docs/performance-tuning.md
Implementation Checklist

落地清单：新增一种非地鼠怪物，出现 10 秒后消失

这个怪物不是地鼠皮肤，而是自己的 entity。它可以复用 ECS、dirty、binding、ViewRegistry 这条链路，但不要塞进 ShrewComponent 里硬凑。

1. 先建自己的 Component

比如 `BonusMonsterComponent`：`type`、`posXRatio`、`posYRatio`、`ageSec`、`durationSec=10`、`visible`、`spawnSeq`。它不是地鼠，就不要混进 `ShrewComponent`。

2. world.ts 给它创建 entity

新增 `createBonusMonsterEntity(world, mapType)`，挂 `BonusMonsterComponent` 和 `DirtyComponent`。如果同屏最多一个，就复用同一个 eid；如果多个，就建池或数组。

3. System 管 10 秒生命周期

`bonusMonsterSystem(deltaSec)` 只改 ECS 字段：出现时 `visible=1, ageSec=0`，每帧累加，超过 10 秒改成 `visible=0` 或进入待复用状态。

4. dirty / binding / view 补齐

`BonusMonsterComponent.visible -> BIT_BONUS_MONSTER_VISIBLE -> DirtyComponent.bonusMonsterDirty -> BonusMonsterViewBinding -> BonusMonsterNode.setVisible()`。

对应清单：world.ts 创建 entity -> System 生命周期 -> DirtyFlags/DirtyAspect -> Binding -> ViewNode -> ViewRegistry -> 测试。
